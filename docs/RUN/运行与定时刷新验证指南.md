# 运行与定时刷新验证指南

- 日期：2026-02-22
- 目的：给出“官方推荐运行方式”与“定时刷新真实运行”的可验证口径（接口 + 指标）。

---

## 1) 官方推荐运行方式（建议）

### Docker（推荐）

使用仓库提供的 `Dockerfile`（内部使用 Gunicorn 启动，默认 `-w 1`，避免多进程重复启动调度器与 session 共享问题）。

关键点：
- 必须设置 `SECRET_KEY`
- 建议设置 `SCHEDULER_AUTOSTART=true`（默认即为 true），用于自动启动后台调度器（心跳 + 定时刷新）
- 持久化挂载 `./data:/app/data`，避免重启后丢数据

### 本地直接运行（开发/调试）

```bash
pip install -r requirements.txt
export SECRET_KEY=xxx
python web_outlook_app.py
```

> 注意：`flask run` 的 reloader 可能导致重复启动；若使用，请确认 `SCHEDULER_AUTOSTART` 行为符合预期。

## 2) 如何验证“定时刷新真的在跑”

### A. 快速健康检查（管理员）

- 接口：`GET /api/system/health`
- 关注字段：
  - `health.database`：数据库是否可用
  - `health.scheduler.enabled`：是否启用定时刷新
  - `health.scheduler.heartbeat_age_seconds`：心跳距离当前的秒数
  - `health.scheduler.healthy`：是否健康（默认阈值：`<= 120s`）
  - `health.refresh.locked / running`：是否存在刷新锁/运行中刷新

### B. 调度器状态与“未来触发预览”

- 接口：`GET /api/scheduler/status`
- 关注字段：
  - `scheduler.enabled / use_cron`：是否启用、调度方式
  - `scheduler.future_runs / next_due`：未来触发时间预览
  - `scheduler.heartbeat_age_seconds`：调度器心跳年龄
  - `refresh.last_scheduled`：最近一次定时刷新（含触发来源、总数/成功/失败、trace_id）
  - `refresh.running`：是否存在运行中的刷新（running 状态）
  - `refresh.lock`：分布式锁状态（用于冲突控制）

### C. 诊断信息（状态一致性/清理策略）

- 接口：`GET /api/system/diagnostics`
- 关注字段：
  - `export_verify_tokens_active`：当前有效导出二次验证 token 数量
  - `login_locked_ip_count`：当前被锁定的 IP 数量
  - `locks`：分布式锁列表
  - `running_runs / last_runs`：刷新运行记录
  - `schema`：数据库升级/版本状态（是否最新、最近一次迁移记录）

### D. 升级过程可验证（数据兼容/升级保障）

- 接口：`GET /api/system/upgrade-status`
- 关注字段：
  - `schema_version / target_version / up_to_date`：当前版本与目标版本
  - `last_trace_id / last_error`：最近一次升级的追踪标识与错误信息（便于定位）
  - `last_migration`：最近一次迁移记录（status/时间/trace_id）
