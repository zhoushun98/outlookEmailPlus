## 1) 运行与定时刷新可靠性

- Web 服务运行能力
  - 探活：`GET /healthz`
  - 登录后系统健康：`GET /api/system/health`
- 后台刷新任务运行能力
  - 调度器心跳：`GET /api/scheduler/status` → `scheduler.heartbeat_age_seconds`
  - 调度器自启动：环境变量 `SCHEDULER_AUTOSTART=true`（见 README/.env.example）
- 官方推荐运行方式（可验证）
  - 推荐：Docker + Gunicorn 单 worker（见 `Dockerfile` 与 `README.md`）
  - 验证口径：见 `docs/RUN/运行与定时刷新验证指南.md`
- 定时刷新能力齐全（启停/两种调度/校验/未来预览）
  - 设置项：`enable_scheduled_refresh`、`use_cron_schedule`、`refresh_interval_days`、`refresh_cron`
  - Cron 校验：`POST /api/settings/validate-cron`
  - 未来预览：`GET /api/scheduler/status` → `scheduler.future_runs / next_due`
- 定时刷新可验证（最近一次/来源/统计）
  - `refresh_runs` 表记录：trigger_source/status/started_at/finished_at/total/success_count/failed_count/trace_id
  - `GET /api/scheduler/status` → `refresh.last_scheduled`
- 刷新冲突控制
  - `distributed_locks` 表作为分布式锁
  - 冲突策略：已有刷新运行时返回冲突提示（409 / skipped）
  - 单测覆盖：`tests/test_distributed_lock.py`

## 3) 错误与追踪一致性

- 统一错误结构 + trace_id
  - `X-Trace-Id` 响应头
  - API error payload：`{code,message,type,status,details,trace_id}`
  - 单测覆盖：`tests/test_error_and_trace.py`（404 structured / trace_id 透传）
- 默认脱敏
  - `sanitize_error_details()` 对 token/password 等脱敏（用于 message/details）
  - `build_error_payload()` 会对 `message` 与 `details` 做默认脱敏
  - 单测覆盖：
    - `tests/test_error_and_trace.py` → `test_sanitize_error_details_masks_tokens`
    - `tests/test_error_and_trace.py` → `test_build_error_payload_sanitizes_message_and_details`
- 多方式回退失败原因聚合（中文可理解）
  - 邮件获取：`GET /api/emails/<email>` 失败时返回 `details.graph/imap_new/imap_old`（前端弹框 `showEmailFetchErrorModal()`）
  - 邮件删除：`POST /api/emails/delete` 全部失败时返回 `error.code=EMAIL_DELETE_ALL_METHODS_FAILED`，并在 `error.details` 汇总 Graph/IMAP 失败原因（同时保留 `details` 字段供进一步排障）
  - 单测覆盖：`tests/test_error_and_trace.py` → `test_delete_emails_all_methods_fail_returns_aggregated_error`
- 未捕获异常统一处理
  - Flask 全局异常处理器：返回稳定错误结构 + trace_id
