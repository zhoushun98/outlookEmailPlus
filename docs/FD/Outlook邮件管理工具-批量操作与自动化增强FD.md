## 3. 后端交付清单

### 3.1 新增 API 接口

#### 3.1.1 验证码和链接提取接口
- **路由**：`GET /api/emails/<email>/extract-verification`
- **功能**：提取指定邮箱最新邮件中的验证码和链接
- **请求参数**：
  - `email`：邮箱地址（路径参数）
- **响应结构**：
  ```json
  {
    "success": true,
    "data": {
      "verification_code": "123456",
      "links": ["https://example.com/verify", "https://example.com/activate"],
      "formatted": "123456 https://example.com/verify https://example.com/activate"
    },
    "message": "提取成功"
  }
  ```
- **失败响应**：
  ```json
  {
    "success": false,
    "error": "未找到验证信息",
    "trace_id": "xxx"
  }
  ```

### 3.2 新增服务模块

#### 3.2.1 验证码提取服务
- **位置**：`outlook_web/services/verification_extractor.py`
- **功能清单**：
  - 获取邮箱最新邮件（同时从收件箱和垃圾邮件获取，按时间排序取最新）
  - 智能验证码识别：
    - 识别关键词（"验证码"、"code"、"验证"、"verification"、"OTP"等）
    - 提取关键词附近的 4-8 位数字/字母组合
  - 保底验证码提取：
    - 如果智能识别失败，提取邮件中所有 4-8 位连续数字/字母组合
    - 取第一个匹配项
  - 链接提取：
    - 使用正则表达式提取所有 HTTP/HTTPS 链接
    - 去重并保持顺序
  - 格式化输出：
    - 验证码和链接用空格分隔
    - 无描述性文字，纯数据输出

### 3.3 数据库变更

#### 3.3.1 settings 表新增配置项
- **配置项清单**：
  - `enable_auto_polling`：是否启用自动轮询（默认：`false`）
  - `polling_interval`：轮询间隔（秒，默认：`10`）
  - `polling_count`：轮询次数（默认：`5`）
- **初始化逻辑**：
  - 在 `init_db()` 函数中添加默认值插入逻辑
  - 使用 `INSERT OR IGNORE` 避免重复插入

### 3.4 更新现有 API

#### 3.4.1 设置 API 更新
- **接口**：`GET /api/settings` 和 `PUT /api/settings`
- **功能**：支持读取和更新轮询配置项
- **响应结构**（GET）：
  ```json
  {
    "success": true,
    "settings": {
      "enable_auto_polling": false,
      "polling_interval": 10,
      "polling_count": 5,
      ...
    }
  }
  ```
- **请求结构**（PUT）：
  ```json
  {
    "enable_auto_polling": true,
    "polling_interval": 30,
    "polling_count": 10
  }
  ```

### 3.5 路由注册

#### 3.5.1 邮件路由更新
- **位置**：`outlook_web/routes/emails.py`
- **新增路由**：
  ```python
  bp.add_url_rule(
      "/api/emails/<email>/extract-verification",
      view_func=impl.api_extract_verification,
      methods=["GET"]
  )
  ```

## 4. 前端交付清单

### 4.1 功能1：全选功能

#### 4.1.1 UI 组件
- **位置**：邮箱面板工具栏，排序按钮右边
- **组件结构**：
  ```html
  <label class="select-all-wrapper">
    <input type="checkbox" id="selectAllAccounts" class="select-all-checkbox">
    <span>全选</span>
  </label>
  ```
- **样式要求**：
  - 与排序按钮保持一致的视觉风格
  - 支持三种状态：未选中、半选中（indeterminate）、全选中
  - 半选中状态使用特殊样式（如横线图标）

#### 4.1.2 交互逻辑
- **全选复选框点击事件**：
  - 如果当前是未选中或半选中状态，选中当前分组下的所有邮箱
  - 如果当前是全选中状态，取消选中所有邮箱
- **邮箱复选框点击事件**：
  - 更新 `selectedAccounts` Set
  - 更新全选复选框状态：
    - 如果所有邮箱都被选中，全选复选框显示为全选中
    - 如果部分邮箱被选中，全选复选框显示为半选中
    - 如果没有邮箱被选中，全选复选框显示为未选中
- **切换分组事件**：
  - 重置全选复选框状态为未选中
  - 清空 `selectedAccounts` Set

#### 4.1.3 状态管理
- **全局变量**：
  - 复用现有的 `selectedAccounts` Set
  - 不需要新增全局变量
- **状态同步**：
  - 全选复选框状态与 `selectedAccounts` 实时同步
  - 批量操作栏显示/隐藏与选中数量联动

### 4.2 功能2：邮件信息快速复制

#### 4.2.1 UI 组件
- **位置**：邮箱列表中，每个账号名称右边
- **组件结构**：
  ```html
  <button class="copy-verification-btn"
          onclick="copyVerificationInfo('email@example.com')"
          title="复制验证信息">
    📋
  </button>
  ```
- **样式要求**：
  - 按钮大小适中，不遮挡账号名称
  - 默认状态：灰色图标
  - 悬停状态：蓝色图标
  - 加载状态：显示加载动画
  - 成功状态：短暂显示绿色对勾（1秒后恢复）

#### 4.2.2 交互逻辑
- **复制按钮点击事件**：
  1. 显示加载状态（禁用按钮，显示加载动画）
  2. 调用后端 API：`GET /api/emails/<email>/extract-verification`
  3. 如果返回成功：
     - 使用 Clipboard API 复制 `formatted` 字段到剪贴板
     - 显示成功提示：右上角弹出"已复制"通知（2秒后消失）
     - 按钮显示绿色对勾（1秒后恢复）
  4. 如果返回失败（未找到验证信息）：
     - 显示失败提示："未找到验证信息"
  5. 如果返回失败（邮箱无数据）：
     - 自动调用 `loadEmails(email, true)` 获取邮件
     - 获取成功后，重新调用提取 API
     - 获取失败后，显示具体错误信息
  6. 恢复按钮状态（启用按钮，隐藏加载动画）

#### 4.2.3 API 调用
- **函数名**：`copyVerificationInfo(email)`
- **功能**：
  - 调用后端 API 提取验证信息
  - 复制到剪贴板
  - 显示成功/失败提示
- **错误处理**：
  - 网络错误：显示"网络错误，请重试"
  - API 错误：显示具体错误信息（从 `error` 字段获取）
  - 剪贴板权限错误：显示"复制失败，请手动复制"

### 4.3 功能3：自动轮询与通知

#### 4.3.1 设置界面
- **位置**：设置页面（`/api/settings` 对应的前端界面）
- **组件结构**：
  ```html
  <div class="setting-group">
    <h3>自动轮询设置</h3>
    <div class="setting-item">
      <label>
        <input type="checkbox" id="enableAutoPolling">
        启用自动轮询
      </label>
    </div>
    <div class="setting-item">
      <label>轮询间隔（秒）：</label>
      <input type="number" id="pollingInterval" min="5" max="300" value="10">
    </div>
    <div class="setting-item">
      <label>轮询次数：</label>
      <input type="number" id="pollingCount" min="1" max="100" value="5">
    </div>
  </div>
  ```
- **交互逻辑**：
  - 加载设置时，从 API 获取当前配置并填充表单
  - 保存设置时，调用 `PUT /api/settings` 更新配置
  - 保存成功后显示提示："设置已保存，下次选中邮箱时生效"

#### 4.3.2 轮询逻辑
- **全局变量**：
  - `pollingTimer`：定时器 ID
  - `pollingCount`：当前轮询次数
  - `lastEmailIds`：上次获取的邮件 ID 列表（用于检测新邮件）
- **轮询启动**：
  - 触发时机：用户在邮箱列表中选中一个邮箱时
  - 前置条件：设置中启用了自动轮询
  - 启动逻辑：
    1. 清除之前的定时器（如果存在）
    2. 重置轮询计数器
    3. 获取当前邮件列表，保存邮件 ID 到 `lastEmailIds`
    4. 启动定时器，按配置的间隔执行轮询
- **轮询执行**：
  - 每次轮询时：
    1. 调用 `loadEmails(currentAccount, true)` 获取最新邮件
    2. 对比新旧邮件 ID 列表，检测是否有新邮件
    3. 如果有新邮件，触发通知
    4. 更新 `lastEmailIds`
    5. 轮询计数器 +1
    6. 如果达到设置的轮询次数，停止轮询
- **轮询停止**：
  - 触发时机：
    - 达到设置的轮询次数
    - 用户切换到其他邮箱
    - 用户关闭页面或退出登录
    - 轮询过程中获取邮件失败
  - 停止逻辑：
    1. 清除定时器
    2. 重置轮询计数器
    3. 清空 `lastEmailIds`

#### 4.3.3 新邮件检测
- **检测逻辑**：
  - 对比新旧邮件 ID 列表
  - 找出新邮件（存在于新列表但不存在于旧列表的邮件）
  - 如果有新邮件，返回新邮件列表
- **函数签名**：
  ```javascript
  function detectNewEmails(oldEmailIds, newEmails) {
    // 返回新邮件列表
  }
  ```

#### 4.3.4 通知组件
- **页面内通知**：
  - **位置**：页面右上角
  - **组件结构**：
    ```html
    <div class="notification-toast" id="emailNotification">
      <div class="notification-icon">📧</div>
      <div class="notification-content">
        <div class="notification-title">新邮件</div>
        <div class="notification-message">user@outlook.com 收到新邮件：您的验证码是 123456</div>
      </div>
      <button class="notification-close" onclick="closeNotification()">×</button>
    </div>
    ```
  - **显示逻辑**：
    - 检测到新邮件时，显示通知
    - 通知内容：邮箱地址 + 邮件主题
    - 5 秒后自动消失，或用户点击关闭按钮
  - **样式要求**：
    - 固定定位在右上角
    - 从右侧滑入动画
    - 半透明背景，白色文字
    - 支持多条通知堆叠显示

- **邮箱列表视觉提示**：
  - **位置**：邮箱名称旁边
  - **组件结构**：
    ```html
    <span class="new-email-badge">●</span>
    ```
  - **显示逻辑**：
    - 检测到新邮件时，在邮箱名称旁边显示红色圆点
    - 用户点击该邮箱并查看邮件后，红点消失
  - **样式要求**：
    - 红色圆点，直径 8px
    - 位置在邮箱名称右侧，垂直居中

## 5. 非功能交付项（必须满足）

### 5.1 性能要求
- 全选操作在 100ms 内完成（即使邮箱数量达到 1000 个）
- 验证码提取 API 响应时间 < 2 秒（不包括获取邮件的时间）
- 轮询操作不影响页面正常使用，不导致页面卡顿

### 5.2 可靠性要求
- 验证码提取成功率 > 90%（基于常见邮件格式）
- 轮询过程中如果出现网络错误或 API 失败，自动停止轮询并提示用户
- 复制操作失败时，给出明确的失败原因和建议操作

### 5.3 兼容性要求
- 全选功能与现有批量操作（移动分组、打标签）完全兼容
- 复制功能支持主流浏览器的 Clipboard API
- 轮询功能不影响现有的手动刷新功能

### 5.4 安全要求
- 验证码提取 API 需要登录验证
- 复制到剪贴板的内容不包含敏感信息（如完整 Token）
- 轮询频率有合理限制，避免过度请求导致账号被封

### 5.5 用户体验要求
- 所有操作都有明确的成功/失败反馈
- 加载状态有明确的视觉提示（加载动画、禁用按钮等）
- 错误提示清晰易懂，告诉用户下一步怎么做

## 9. 风险与依赖

### 9.1 技术风险
- **验证码识别准确率**：不同邮件格式差异较大，可能导致识别失败
  - 缓解措施：提供保底提取规则，并支持用户手动查看邮件
- **浏览器剪贴板权限**：部分浏览器可能限制剪贴板访问
  - 缓解措施：使用标准 Clipboard API，失败时提示用户手动复制
- **轮询性能影响**：高频轮询可能影响页面性能
  - 缓解措施：设置合理的默认值，使用异步请求

### 9.2 依赖项
- 依赖现有的邮件获取 API（Graph API / IMAP）
- 依赖现有的批量操作功能
- 依赖现有的通知组件（可复用或新建）
- 依赖浏览器 Clipboard API
