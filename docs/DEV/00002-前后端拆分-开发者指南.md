# 00002｜前后端拆分与模块化：开发者指南

- 日期：2026-02-23
- 目的：说明本轮“前后端拆分与模块化”的**目录结构、模块职责、依赖方向与新增 API 的推荐路径**，便于持续演进且不破坏对外契约。
- 约束：保持 **URL/响应契约/错误结构/trace_id/调度器行为** 不回归；前端保持**零构建**。

---

## 1. 总体结构速览

- 稳定部署入口：`web_outlook_app:app`（`web_outlook_app.py`）
- 后端包：`outlook_web/`（Flask + SQLite）
- 前端：`templates/`（Jinja 模板）+ `static/`（CSS/JS，零构建）
- 测试：`tests/`（统一 `unittest` 门禁）

---

## 2. 后端模块职责（outlook_web）

### 2.1 入口与装配

- `web_outlook_app.py`
  - 对外保持 `web_outlook_app:app` 不变（部署/Gunicorn/Docker 兼容）
  - 迁移期仍 re-export legacy 的函数/常量访问路径，但**最终导出的 `app` 必须是工厂装配后的 app**
- `outlook_web/app.py`
  - `create_app()`：集中装配（DB、CSRF、错误处理、Blueprint 注册、调度器启动控制）

### 2.2 Routes（Blueprint）

- `outlook_web/routes/*`
  - 仅负责“URL → view_func 映射”
  - 迁移期 view_func 复用 `outlook_web/legacy.py` 中的实现，确保 URL/响应结构不变

### 2.3 兼容层（迁移期）

- `outlook_web/legacy.py`
  - 迁移期的主要 view 函数实现（以及少量兼容导出）
  - 目标：逐步把“SQL/业务编排/通用能力”迁出到更小的模块，legacy 只保留薄薄的控制层

### 2.4 分层：repositories / services

- `outlook_web/repositories/*`
  - 负责 SQL 与数据映射（尽量不掺杂业务编排）
- `outlook_web/services/*`
  - 负责业务编排与多后端回退（Graph/IMAP/GPTMail/刷新编排/删除回退聚合等）

### 2.5 横切能力（基础设施）

- `outlook_web/config.py`：环境变量/默认值集中
- `outlook_web/errors.py`：统一错误结构、`trace_id`、默认脱敏
- `outlook_web/db.py`：DB 连接管理、`init_db()` 与迁移逻辑
- `outlook_web/security/*`：鉴权/加密/CSRF
- `outlook_web/audit.py`：审计写入与查询

---

## 3. 推荐依赖方向（避免循环依赖）

建议依赖方向（由上到下）：

`routes` → `legacy(view/controller)` → `services` → `repositories` → `db`

横切模块（`config/errors/security/audit`）可被上层引用，但应避免反向依赖到 `routes`。

---

## 4. 如何新增/调整一个 API（推荐流程）

1. **确定领域**：落在 `outlook_web/routes/<domain>.py`（若是新领域再新增一个 blueprint 文件）
2. **先写/改测试**（至少满足其一）：
   - 抽样契约门禁：`tests/test_smoke_contract.py`（保证 URL/契约不回归）
   - 边界/回归单测：按风险点新增（例如错误聚合、脱敏、回退策略）
3. **实现顺序（推荐）**：
   - repo：`outlook_web/repositories/*`（SQL/映射）
   - service：`outlook_web/services/*`（业务编排）
   - view：迁移期先在 `outlook_web/legacy.py` 写一个“薄控制层”函数（鉴权/参数解析/调用 service）
4. **路由绑定**：
   - 在 `outlook_web/routes/<domain>.py` 用 `bp.add_url_rule(...)` 绑定 URL（保持 URL/方法/响应结构不变）
5. **验证门禁**：
   - `python -m unittest discover -s tests -v`

## 5. 前端结构（零构建）

- 模板：`templates/index.html`
  - 只保留页面骨架与结构（不再堆叠内联 CSS/JS）
- 静态资源：
  - `static/css/main.css`
  - `static/js/main.js`：全局状态、CSRF、初始化与通用逻辑
  - `static/js/features/*.js`：按 feature 拆分的脚本文件（仍保持全局函数，兼容内联 `onclick` 调用）

建议新增前端功能时：

- 优先在 `static/js/features/` 新增对应文件或扩展现有 feature 文件
- 保持函数命名清晰、避免隐式全局变量（必要时显式挂到 `window`）
- 不引入构建链（不要求 bundler）

---

## 6. 测试与本地运行

- 运行单测：`python -m unittest discover -s tests -v`
- 测试导入入口：`tests/_import_app.py`
  - 注入 `SECRET_KEY / LOGIN_PASSWORD / DATABASE_PATH`
  - 禁用调度器自启动（避免测试期间启动后台线程）

---

## 7. 常见坑（本轮已踩过）

- **`web_outlook_app.py` 的 `from outlook_web.legacy import *` 会把 legacy 的 `app` 覆盖掉**：必须确保最终导出的 `app` 绑定到 `create_app()` 的实例，否则 Blueprint 端点（如 `pages.login`）会缺失并导致运行/测试失败。
- CSRF：测试场景通常会关闭 CSRF（见 `tests/_import_app.py`），生产建议开启（依赖 `flask-wtf`）。
- 错误结构与 `trace_id`：新增 API 时应沿用统一错误 payload 与 `X-Trace-Id` 透传策略，避免前端展示回归。
