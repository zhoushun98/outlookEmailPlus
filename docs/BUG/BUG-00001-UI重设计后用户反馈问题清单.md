## 二、问题详细分析

### BUG-001: 临时邮箱创建后列表不显示邮箱

**类型：** 缺陷  
**严重度：** 🔴 HIGH — 核心功能无法使用  
**复现步骤：**
1. 导航到「临时邮箱」页面
2. 点击「创建邮箱」按钮
3. 观察：API 调用成功但列表不显示新创建的邮箱

**根因分析：**
- `generateTempEmail()` 在 `temp_emails.js` 中调用 `/api/temp-emails` 创建邮箱
- 创建成功后调用 `loadTempEmails()` 刷新列表
- `renderTempEmailList()` 渲染卡片到 `#tempEmailContainer`
- 需要验证：API 返回数据格式是否正确、渲染函数是否匹配新容器 ID

**修复方案：**
- 检查 `loadTempEmails()` 是否正确将数据传给 `renderTempEmailList()`
- 确保 `renderTempEmailList()` 向 `#tempEmailContainer` 写入 HTML
- 添加调试日志确认数据流

**影响范围：** `static/js/features/temp_emails.js`

### BUG-002: 选中账号名称未在邮件栏顶部显示

**类型：** 回归  
**严重度：** 🔴 HIGH — 关键 UX 功能丢失  
**复现步骤：**
1. 在「账号管理」页面选择一个分组
2. 点击一个邮箱账号
3. 观察：邮件列表栏顶部没有显示当前选中的邮箱地址

**根因分析（已确认）：**
- `selectAccount()` (accounts.js 第 13 行) 执行 `document.getElementById('currentAccount').classList.add('show')`
- `#currentAccount` 是 `<span>` 元素，添加 `.show` 类无效果
- 父级 `#currentAccountBar` 保持 `display:none`，永远不会显示
- **核心问题：应该操作 `#currentAccountBar` 的 `display` 属性，而非子元素的 class**

**修复方案：**
```javascript
// accounts.js selectAccount() 中：
// 旧代码（错误）：
document.getElementById('currentAccount').classList.add('show');
// 新代码（修复）：
document.getElementById('currentAccountBar').style.display = '';
```

**影响范围：** `static/js/features/accounts.js` 第 13 行

### BUG-003: 系统设置页面需要额外点击才能看到设置

**类型：** 回归  
**严重度：** 🟡 MEDIUM — UX 不便  
**复现步骤：**
1. 点击侧边栏「系统设置」
2. 观察：页面显示占位文字「请使用设置弹窗配置系统参数」和一个「打开设置」按钮
3. 需要再点击按钮才能看到设置内容

**根因分析：**
- `#page-settings` (index.html 第 282-292 行) 是占位内容
- `navigate()` 函数中没有 `if (page === 'settings') loadSettings();` 的调用
- 设置内容全部在 `settingsModal` 弹窗中，未内联到页面

**修复方案（两种选择）：**
1. **方案 A（推荐）：** 将设置内容直接渲染到 `#page-settings` 页面内，导航到 settings 时调用 `loadSettings()` 并将内容写入页面
2. **方案 B：** 导航到 settings 时自动打开设置弹窗 `showSettingsModal()`

**影响范围：** 
- `templates/index.html` `#page-settings` 区域
- `static/js/main.js` `navigate()` 函数 + `loadSettings()`

### BUG-004: 弹窗（Modal）偏向左边

**类型：** 缺陷  
**严重度：** 🟡 MEDIUM — 视觉问题  
**复现步骤：**
1. 打开任意弹窗（如添加分组、添加账号等）
2. 观察弹窗是否水平居中

**根因分析：**
- CSS `.modal` 使用 `display: none; align-items: center; justify-content: center;`
- `.modal.show { display: flex; }` 激活 flex 布局
- `.modal-content` 使用 `max-width: 560px; width: 100%;`
- 理论上应该居中，但需要实际浏览器验证
- 可能的原因：
  1. 某些弹窗内的 `modal-content` 有额外的 margin/padding
  2. 侧边栏偏移导致视觉上不居中（modal 使用 `position:fixed; inset:0;` 覆盖全屏，不受侧边栏影响）
  3. 某些弹窗可能有内联样式覆盖

**修复方案：**
- 确认 `.modal` 和 `.modal-content` 的样式链无冲突
- 为 `.modal-content` 添加 `margin: auto;` 作为额外保险
- 检查 `modals.html` 中是否有内联样式干扰

**影响范围：** `static/css/main.css` Modal 区域

### BUG-010: 仪表盘分组概览统计数据不准确

**类型：** 缺陷  
**严重度：** 🟡 MEDIUM  
**复现步骤：**
1. 导航到仪表盘
2. 查看分组概览中的邮箱数/目录数
3. 与实际数据对比

**根因分析：**
- `loadDashboard()` (main.js 第 129-188 行) 遍历所有分组
- 对每个分组调用 `/api/accounts?group_id=` 获取账号列表
- 统计 `totalAccounts`, `validTokens`, `expiredTokens`
- 可能的问题：
  1. API 返回的账号列表不完整
  2. Token 状态判断逻辑有误（`a.token_status === 'valid'` 可能不匹配实际字段名）
  3. 分组概览中的「目录数」含义不明确

**修复方案：**
- 验证 API 返回数据结构，确认 `token_status` 字段名
- 检查是否遗漏了没有分组的账号
- 完善统计逻辑，添加分组数量统计

**影响范围：** `static/js/main.js` `loadDashboard()` 函数

### BUG-011: 临时邮箱应有独立页面和分组系统

**类型：** 增强  
**严重度：** 🟢 LOW  
**描述：** 用户希望临时邮箱不再作为分组的子项，而是完全独立的页面，有自己的分类/分组系统。

**当前状态：**
- 临时邮箱已有独立页面 `#page-temp-emails`
- 但页面内只是简单的卡片列表 + 创建按钮
- 用户希望有类似邮箱管理的三栏布局（分类/邮箱列表/邮件内容）

**修复方案：**
- 重设计 `#page-temp-emails` 页面为两栏布局：
  - 左栏：临时邮箱列表（带创建/删除功能）
  - 右栏：选中邮箱的收到的邮件列表和详情
- 后端 API 已支持 `/api/temp-emails/<email>/messages` 获取邮件

**影响范围：**
- `templates/index.html` `#page-temp-emails` 区域
- `static/js/features/temp_emails.js` 大幅改造
- `static/css/main.css` 新增临时邮箱两栏布局样式
