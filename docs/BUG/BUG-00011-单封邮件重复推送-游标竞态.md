# BUG-00011 单封邮件被重复推送多次（游标竞态）

> 关联功能: TODO-00007 / FD-00007
> 前置修复: BUG-00007 ~ BUG-00010
> 分析时间: 2026-03-05

---

## 一、问题概述

| 编号 | 标题 | 严重程度 | 类型 |
|------|------|---------|------|
| **D-1** | 同一封邮件在 Telegram 被推送多次 | 🔴 高 | 游标逻辑 |

---

## 二、现象

用户在 Telegram 收到同一封邮件的重复通知。邮件内容完全相同，但在不同的 Job 周期中被推送了 2 次或更多次。

---

## 三、根因分析

### 3.1 游标设定逻辑缺陷

**修复前**，游标始终设为 `job_start_time`（任务启动的 UTC 时间），而非已处理邮件的最大 `received_at`：

```python
# 修复前 — telegram_push.py:400
update_telegram_cursor(account["id"], job_start_time)
```

### 3.2 竞态时间线

```
Job N:
  job_start_time = 2026-03-05T13:02:00 (本地 UTC 时间)
  fetch → 发现 email(received_at = 2026-03-05T13:02:03)  ← 邮件服务器时钟快 3s
  推送成功 ✓
  cursor = 2026-03-05T13:02:00  ← BUG：应为 13:02:03

Job N+1:
  cursor = 2026-03-05T13:02:00
  fetch: receivedDateTime > 2026-03-05T13:02:00
  email(13:02:03) > cursor(13:02:00) → 再次拉取 → 重复推送 ✗
```

### 3.3 触发条件

| 条件 | 说明 |
|------|------|
| 邮件服务器时钟快于本地 | Graph API 的 `receivedDateTime` 由 Microsoft 服务器设定 |
| 邮件在 fetch 窗口期间到达 | fetch 耗时 4-6 秒，此期间到达的邮件 received_at > job_start_time |
| IMAP INTERNALDATE 时钟偏差 | 与 IMAP 服务器时钟不同步时同样触发 |

### 3.4 与 MailAggregator_Pro 对比

| 维度 | 我们的实现 | MailAggregator_Pro |
|------|-----------|-------------------|
| 防重机制 | 时间游标 | **Message-ID 去重**（数据库唯一约束） |
| 游标策略 | ❌ 固定 `job_start_time` | ✅ 无游标依赖（基于 Message-ID） |
| 时钟偏差容忍 | ❌ 零容忍 | ✅ 天然免疫 |

MailAggregator_Pro 使用 `EmailRecord.message_id` 唯一索引 + `INSERT OR IGNORE` 实现幂等，不依赖时间对比。

---

## 四、修复方案

### 4.1 核心修复：游标取 max(job_start_time, max_received_at)

```python
# 修复后 — telegram_push.py
max_received = max(
    (em.get("received_at", "") for em in emails),
    default="",
)
new_cursor = max(job_start_time, max_received) if max_received else job_start_time
update_telegram_cursor(account["id"], new_cursor)
```

**效果**：
- 邮件 `received_at > job_start_time` → 游标推进到 `received_at`，下轮不再拉取
- 邮件 `received_at < job_start_time` → 游标 = `job_start_time`，行为不变

### 4.2 修改范围

| 文件 | 改动 |
|------|------|
| `outlook_web/services/telegram_push.py` | `run_telegram_push_job` 游标计算逻辑（+6 行） |
| `tests/test_telegram_push.py` | 新增 `TestCursorClockSkew`（3 个测试） |

## 六、P2 实施：Message-ID 去重（已完成 ✅）

### 6.1 设计

参考 MailAggregator_Pro 的 `EmailRecord.message_id` 唯一约束方案，在游标防重之上增加第二层防线。

| 层 | 机制 | 防御场景 |
|----|------|---------|
| L1 | 时间游标 `max(job_start_time, max_received_at)` | 时钟偏差、fetch 窗口 |
| L2 | `telegram_push_log` Message-ID 唯一约束 | 极端场景（游标被外部修改、数据库回滚等） |

### 6.2 数据库变更

**Schema v5** — 新增 `telegram_push_log` 表：

```sql
CREATE TABLE telegram_push_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id INTEGER NOT NULL,
    message_id TEXT NOT NULL,
    pushed_at TEXT NOT NULL,
    UNIQUE(account_id, message_id)
);
```

### 6.3 代码变更

| 文件 | 变更 |
|------|------|
| `outlook_web/db.py` | Schema v4 → v5，新增 telegram_push_log |
| `outlook_web/services/telegram_push.py` | `_is_message_pushed()`, `_record_pushed_message()`, `_cleanup_push_log()`，IMAP 提取 Message-ID header，Graph 提取 internetMessageId |
| `tests/test_telegram_push.py` | +4 TestMessageIdDedup，更新 `_make_email` 包含 message_id |

### 6.4 Message-ID 提取策略

| 邮件源 | 提取方式 | 回退 |
|--------|---------|------|
| IMAP | `Message-ID` / `Message-Id` header | `imap:{email}:{uid}` |
| Graph API | `internetMessageId` 字段 | Graph `id` 字段 |

### 6.5 清理策略

`PUSH_LOG_RETENTION_DAYS = 7` — 每次 Job 结束自动清理超过 7 天的记录，防止表无限增长。
